# Cross-Agent Harness

Codex と Claude Code が同じリポジトリで安全に作業するための共通ルール。

## 基本方針

- 作業開始時に `CLAUDE_CODE_HANDOFF.md` の最新セクションを読む。
- プロジェクト固有の判断は `.claude/rules/project-collaboration-profile.md` を優先する。
- 既存の未コミット変更を自分の変更として扱わない。重なる場合は止めて確認する。
- 実装・レビュー・検証の担当境界を handoff に明記する。
- merge / publish はユーザーの明示指示なしに進めない。

## 作業開始（実装・レビュー担当）

1. `git status --short --branch` で worktree を確認する。
2. `CLAUDE_CODE_HANDOFF.md` の最新セクションから目的、触ってよい範囲、検証方法を読む。
3. 仕様が曖昧な場合は、推測で広げず完成条件を確認する。
4. **handoff に書かれたパス・IPC 契約・削除対象に不足があれば、開始前に依頼者へ確認する**。実体ファイル名と齟齬がある場合（過去事例: db/index.js → 実体 schema.js、channelsRepository → 実体 channelRepository）は依頼者の意図を確認してから進める。

## 依頼起票（設計担当）

1. **handoff に書くファイルパスは実体で確認する**。ls / grep で存在を確認してから記述する。
2. **IPC contract を含む依頼は 4 点対称で記述する**。`main handler` / `preload exposure` / `renderer 呼び出し` / `event 発火・購読のペア` のすべてを完成条件に列挙する。event ペアが該当しない request/response IPC は `N/A と理由` を明記する。
3. **コンポーネント・ファイル置き換え時は「削除すべきファイル」を完成条件に明記する**。新規ファイル作成と旧ファイル削除を 1 つの依頼内で扱う。
4. **UX 大規模変更は事前にモック・ASCII 図・参考スクショなど低コスト案を依頼者と共有してから依頼起票する**。実装後の手戻りを防ぐ。

## 作業中

- 触るファイルを最小化する。
- 相手エージェントの変更を巻き戻さない。
- 設計判断、検証結果、未解決リスクは handoff に残す。
- セキュリティ、認可、個人情報、外部 API quota、公開判断はプロジェクト profile の重大指摘に従う。
- **UX 大規模変更**（タブ統合・新規ハブコンポーネント等）は実装完了直後にスクリーンショットや短い動画で依頼者に共有し、想定とのズレを早期に検出する。事前のモック共有（依頼起票セクション参照）と合わせて二段で確認する。

## 完了条件

- 完成条件を満たす。
- profile の verify コマンドを実行し、結果を handoff に記録する。
- 実行できない検証は理由と代替確認を記録する。
- 反対側レビューが必要な場合は、レビュー観点を明示して依頼できる状態にする。
