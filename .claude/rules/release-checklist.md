# リリースチェックリスト

リリース時は以下を **この順番で** 実施する。スキップ禁止。

---

## 1. 事前検証（必須）

`/verify` を実行して lint・test・build がすべてパスすることを確認する。
エラーがある場合はリリース作業に進まない。

---

## 2. README 機能一覧の更新

`README.md` の「機能」セクションを今回のリリース内容に合わせて更新する。

**更新が必要なケース：**

- 新しいタブ・画面を追加した
- ユーザーが操作する新機能を追加した（ボタン・フィルター・検索等）
- 既存機能の動作が変わった

**更新不要なケース：**

- バグ修正のみ
- 内部リファクタリング・テスト追加のみ

---

## 3. リリースノートの作成

`gh release edit vX.X.X --notes "..."` で公開前に書く（draft 状態のうちに）。

### リリースノートの構成

```
## vX.X.X — <リリースの主題を一言で>

### 新機能
（追加された機能をユーザー視点で箇条書き。技術用語は避ける）

### バグ修正
（修正した問題を「〇〇が〇〇になっていた問題を修正」形式で記載）

### データ移行（DB変更があれば）
（「初回起動時に自動マイグレーションが実行されます」等）
```

**品質基準：**

- ユーザーが「これで何が変わったか」を 30 秒で理解できること
- コミットメッセージのコピペではなく、ユーザー視点で再表現する
- 「内部処理を改善」「リファクタリング」などの技術内部は書かない

---

## 4. X（Twitter）広告テキストの生成

リリース完了後に以下のフォーマットで生成し、ユーザーに提示する。
**投稿はユーザーが手動で行う**ためコピーしやすい形で出力すること。

### フォーマット

```
{アプリの価値を1文で} を v{バージョン} にアップデート {絵文字}

{今回の変更点をカジュアルに1〜2文。技術用語なし}

{ターゲットへの一言} 👇
https://github.com/harness17/youtube-schedule

#個人開発 #Electron #YouTube #個人開発者と繋がりたい
```

### ルール

- 本文 + URL + ハッシュタグ合計 **140文字以内**（URL は 23文字換算）
- ハッシュタグは 4つ固定（`#個人開発 #Electron #YouTube #個人開発者と繋がりたい`）
- 「バグを修正しました」より「〇〇が正しく動くようになりました」と **メリット形式** で書く
- 新機能がある場合は機能名より **解決する悩み** を前面に出す
- バグ修正のみのリリースは「地味だけど大事な修正です」など正直に書いてよい

### 例

```
YouTube配信スケジューラー を v1.5.0 にアップデート 🔔

見逃したくない配信に🔔を立てると終了後に自動でリスト化。
アーカイブ・お気に入りタブも追加しました。

YouTubeの配信見逃しがちな人に刺さりそう 👇
https://github.com/harness17/youtube-schedule

#個人開発 #Electron #YouTube #個人開発者と繋がりたい
```

## 5. develop との同期

master への変更（README 更新など）を develop にも反映する。

```bash
git checkout develop
git merge --no-ff master -m "chore: sync develop with master (vX.X.X)"
git push origin develop
git checkout master
```
